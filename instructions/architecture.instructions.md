# Architecture Instructions

## 設計原則

- 責務を明確に分離する
- UI と Backend の契約（API インターフェース）を先に定義する
- データフローを明示する
- エラー処理方針を統一する

## 設計ドキュメント

設計書は `docs/design/<feature>-architecture.md` に出力する。

### 設計書に含めるもの

1. **システム構成**: コンポーネント図、責務分割
2. **影響範囲**: 既存機能への影響
3. **UI / Backend 契約**: API エンドポイント、リクエスト/レスポンス型
4. **データフロー**: シーケンス図またはフロー図
5. **エラー処理**: エラーコード体系、リトライ方針
6. **テスト観点**: テスト可能な境界
7. **セキュリティ影響**: 認証/認可、データ保護
8. **運用影響**: モニタリング、ログ、アラート
9. **リスク**: 技術的リスクと緩和策

## レビュー観点

- 責務が正しく分離されているか
- 既存アーキテクチャとの整合性があるか
- スケーラビリティ上の問題がないか
- セキュリティ上の懸念がないか
